## 一、项目定位

### 1.1 这是什么项目？

**New API** 是一个 AI API 网关/代理平台。打个比方：

> 市面上有几十家 AI 厂商（OpenAI、Claude、Gemini……），每家都有自己的接口、计费方式、Key 格式。  
> **New API 就像一个"万能转接头"** — 你只需要接入这一个平台，就能用上所有厂商的 AI 服务，统一管理 Key、统一计费、统一查看用量。

**前端**就是这个平台的"门面"——用户看到的所有页面、点击的所有按钮、填写的所有表单，都是前端负责的。

### 1.2 用户是谁？

| 用户角色 | 典型场景 | 关心的功能 |
|---------|---------|-----------|
| 🧑‍💻 **开发者** | 接入 AI API 做开发 | 获取 Key、看文档、查用量日志 |
| 🏢 **管理员** | 运营这个平台 | 管用户、管渠道、设价格、看数据 |
| 💰 **普通用户** | 充值买额度、用 AI | 钱包充值、模型选购、聊天测试 |
| 🌐 **访客** | 了解这个平台是干嘛的 | 首页、定价页、关于页 |

### 1.3 技术栈（已有）

| 类别 | 技术选型 | 为什么选它 |
|------|---------|-----------|
| 框架 | React 19 + TypeScript | 主流、生态好、类型安全 |
| 构建 | Rsbuild | 比 Vite 更快，Rspack 生态 |
| 路由 | TanStack Router | 类型安全路由、文件系统路由 |
| 数据请求 | TanStack Query + Axios | 缓存、重试、乐观更新 |
| 状态管理 | Zustand | 轻量、无 boilerplate |
| UI 组件 | Base UI + 自建 shadcn 风格组件 | 无样式预设、完全可控 |
| 样式 | Tailwind CSS v4 | 原子化 CSS、设计系统一致 |
| 表单 | React Hook Form + Zod | 高性能表单 + 类型安全校验 |
| 图表 | Recharts + VChart | 轻量图表 + 复杂可视化 |
| 国际化 | i18next | 6 种语言支持 |
| 图标 | Hugeicons + Lucide | 现代图标库 |
| 包管理 | Bun | 比 npm 快 10x+ |

## 三、设计系统（Design System）

> 这是前端"长什么样"的总规范。所有新增页面、组件都必须遵守，确保整个平台看起来像「同一个人设计的」。

### 3.1 设计原则

| 原则 | 解释 | 反例 |
|------|------|------|
| 🧩 **一致性** | 同样的东西，放哪里都长一样 | 这个页面的按钮是蓝色，那个是绿色 |
| 📐 **层次感** | 重要信息一眼看到，细节按需展开 | 所有字一样大，看不出来重点 |
| ⚡ **即时反馈** | 每次操作都有响应（loading / 成功 / 失败） | 点了按钮啥反应都没有 |
| ♿ **无障碍** | 键盘能操作、屏幕阅读器能读 | 只能用鼠标，色盲看不懂 |
| 📱 **移动优先** | 先考虑手机上的体验，再适配大屏 | 手机上字挤成一团 |

### 3.2 颜色体系

```
主题色（品牌色）
├── primary      主色（按钮、链接、强调）
├── secondary    辅色（次要按钮、标签）
│
语义色（传递信息）
├── success      成功 / 绿色
├── warning      警告 / 黄色
├── danger       危险 / 红色
├── info         提示 / 蓝色
│
中性色（背景、文字、边框）
├── background   页面背景
├── foreground   主要文字
├── muted        次要文字 / 灰色
├── border       边框 / 分割线
│
暗色模式：以上所有颜色都有对应的 dark 版本
```

### 3.3 字体层级

```
标题（Heading）
├── h1    页面大标题     ~2rem / 32px   粗体
├── h2    区块标题       ~1.5rem / 24px  粗体
├── h3    子区块标题     ~1.25rem / 20px 中等
├── h4    卡片内标题     ~1rem / 16px    中等
│
正文（Body）
├── body-lg   大段正文    ~1rem / 16px
├── body      默认正文    ~0.875rem / 14px
├── body-sm   小字/辅助   ~0.75rem / 12px
│
等宽字体（代码、Key、Token）
└── mono      API Key、代码块   等宽字体
```

### 3.4 间距规范

```
基础单位：4px（Tailwind 的默认单位）

常用间距：
├── 4px   (p-1)   极小间距（图标和文字之间）
├── 8px   (p-2)   紧凑间距（表单标签和输入框之间）
├── 12px  (p-3)   常规内边距（卡片内边距）
├── 16px  (p-4)   标准外边距（区块间距）
├── 24px  (p-6)   大间距（页面区块分隔）
├── 32px  (p-8)   超大间距（页面级留白）
└── 48px+ (p-12+) 页面上下留白
```

### 3.5 组件规范速查

| 组件 | 使用场景 | 关键规范 |
|------|---------|---------|
| **Button** | 主要操作（提交、确认） | 3 种变体：primary / secondary / ghost；3 种尺寸：sm / default / lg |
| **Input** | 文本输入 | 有 label、有 placeholder、有错误提示 |
| **Select** | 下拉选择 | 超过 8 个选项用搜索；选项分组用分隔线 |
| **Dialog/Modal** | 确认操作、表单填写 | 有标题、有取消按钮、点遮罩可关闭 |
| **Toast** | 操作结果反馈 | 成功绿、失败红、3 秒自动消失 |
| **Table** | 数据列表展示 | 有表头、支持排序、空状态提示 |
| **Card** | 信息分组展示 | 有标题、有边框或阴影、内边距一致 |
| **Tabs** | 内容分类切换 | 当前 tab 高亮、切换无闪烁 |
| **Empty State** | 无数据时展示 | 图标 + 一句话说明 + 引导操作按钮 |
| **Skeleton** | 数据加载中 | 灰色占位块 + 呼吸动画 |
| **Spinner** | 操作处理中 | 居中旋转动画 |

## 五、注意事项 & 风险清单

> ⚠️ 以下每一项都是实际开发中容易踩的坑，提前知道可以避免大量返工。

### 5.1 设计类注意事项

| # | 注意事项 | 为什么重要 | 怎么避免 |
|---|---------|-----------|---------|
| 1 | **不要在所有地方用同一种字号** | 用户分不清哪些是标题、哪些是正文、哪些是辅助信息 | 严格遵循字体层级（见 §3.3） |
| 2 | **不要用纯黑色（#000）** | 现实中不存在纯黑色，用深灰更自然 | 正文用 `#1a1a2e` 系、标题用 `#0f0f1a` |
| 3 | **暗色模式不是简单的反色** | 纯白文字在黑底上刺眼 | 暗色模式文字用灰白（如 `#e0e0e0`），降低对比度到舒适范围 |
| 4 | **按钮要区分主次** | 一个页面上 5 个蓝色按钮，用户不知道该点哪个 | 最多 1 个 primary + N 个 secondary/ghost |
| 5 | **移动端表格不要硬塞** | 手机上 8 列表格完全没法看 | 用卡片布局替代，或只显示关键列 + "展开详情" |
| 6 | **Loading 时间长于 300ms 就要给反馈** | 用户以为卡住了，疯狂点击 | Skeleton 占位 + Spinner（见 §3.5） |
| 7 | **颜色不能是唯一的信息传递方式** | 色盲/色弱用户看不懂红绿区分 | 错误状态加图标 ❌、成功状态加图标 ✅ |

### 5.2 技术类注意事项

| # | 注意事项 | 为什么重要 | 怎么避免 |
|---|---------|-----------|---------|
| 1 | **Bun 没装** | 当前环境 `bun` 不在 PATH 里，但前端必须用 bun | 先装 Bun：`powershell -c "irm bun.sh/install.ps1 | iex"` |
| 2 | **每次改 TS/TSX 后跑 typecheck** | AGENTS.md 明确规定 | `bun run typecheck`，有错就修 |
| 3 | **i18n 不能忘** | 所有面向用户的文案都要走 `t()` | 写新文案时先在 i18n 文件里加 key |
| 4 | **暗色模式兼容** | 很多人晚上用暗色模式 | 每个新页面都在暗色模式下过一遍 |
| 5 | **大列表性能** | 100+ 条数据渲染会卡 | 用 `@tanstack/react-virtual` 虚拟滚动 |
| 6 | **API 请求去重** | 同一个请求可能同时发多次 | 用 TanStack Query 的 `queryKey` + `staleTime` |

### 5.3 流程类注意事项

| # | 注意事项 | 为什么重要 |
|---|---------|-----------|
| 1 | **先画原型再写代码** | 用 Figma / Excalidraw / 手绘草图确认布局，比写完了再改快 10 倍 |
| 2 | **一个 PR 只做一件事** | 改按钮 + 改表单 + 改颜色混在一起，出问题没法回滚 |
| 3 | **改完立刻在看（真机 + 手机）** | 开发环境看起来正常，实际设备上可能炸了 |
| 4 | **改 UI 前先截图旧版** | 方便对比，也让 reviewer 一眼看出改了什么 |
| 5 | **关键操作要二次确认** | 删除、批量操作不可逆，必须有确认弹窗 |
